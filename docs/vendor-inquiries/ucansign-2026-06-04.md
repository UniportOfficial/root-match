## 1. 인사 + 발신처 소개

안녕하세요, UCanSign 운영팀 귀중.

저희는 동국대학교 기술창업캡스톤디자인 과목에서 진행 중인 **루트매칭 (Rootmatching)** 팀입니다. 한국 뿌리산업(주조 · 금형 · 소성가공 · 용접 · 표면처리 · 열처리 6대 공정)의 B2B 발주처와 공장 간 매칭 + 전자계약 + 에스크로 결제까지 한 번에 처리하는 SaaS를 개발하고 있습니다.

전자계약 vendor 결정 과정에서 UCanSign Public API 가이드를 검토한 결과:

- **PRD FR-5 (계약) P0 4/5 cover** + audit-trail PDF 지원
- **Webhook 4 events 명시 지원** (sign_creating / signing_canceled / signing_completed / signing_completed_all)
- **임베딩 페이지 4 endpoints** — iframe 통합으로 자사 화면 안에서 서명 가능
- **가격 투명**: 1건당 100원 + 신규 10건 무료 포인트
- **customValue × 6 필드**: 발신측 internal IDs 매핑 가능 → multi-tenant 매핑 용이

위 강점을 종합하여 **UCanSign을 정식 채택**하기로 결정했습니다.

Phase 3(전자계약 통합) 본격 구현 전, 가이드에 명시되지 않은 몇 가지 운영/기술 항목을 확인드리고자 본 문의를 드립니다.

## 2. 통합 컨텍스트 (가이드 기준 저희 측 정리)

- **Base URL**: `https://app.ucansign.com/openapi`
- **인증**: API KEY → accessToken (30분 만료) → `Authorization: Bearer <token>`
- **token 캐시 전략**: 만료 5분 전 자동 재발급 (`UCanSignAuthService` 클래스 + Redis 공유 캐시 horizontal scale 시)
- **사용 예정 endpoint**: `/templates/{id}` (탬플릿 기반 서명문서 생성) → `/embedding/sign-creating` 또는 자동 발송 → `/webhooks/ucansign` 수신 → `/documents/{id}/file` + `/audit-trail`
- **2차인증**: prod default `mobile_identification` (휴대전화 실명인증) 양측 + factory 측에 `mobile_otp` 옵션
- **통합 형태**: NestJS `contracts/` 모듈 + `UCanSignClient` HTTP wrapper + Prisma `Contract` model + webhook controller (4 events)

## 3. ⚠️ 보안 사항 — API KEY 발급 + 보관

UCanSign 채택 결정 후 정식 가입 + API KEY 발급 단계입니다.

**요청 사항**:

1. **개발자센터 가입 + API KEY 발급 절차** 안내 부탁드립니다.
2. **dev/prod 키 분리 발급 가능 여부** (개발자센터에서 별도 키 2개 발급, 별도 포인트 한도 등). 자세한 dev 환경 정책은 §4 Q2에서 별도로 문의드립니다.
3. (참고) 저희는 키를 GitHub Actions repo secret + 운영 환경 별도 secret (Railway/Fly.io)으로만 저장합니다. chat/IDE/git에 plain text 노출이 없도록 운영 권고 정책이 vendor 측에 있으면 안내 부탁드립니다.

## 4. Tier 1 — CRITICAL 문의 (Phase 3 진입 blocker)

다음 4개 항목은 답변 받기 전까지 Phase 3 본격 구현이 보류됩니다. **우선 회신 부탁드립니다.**

### Q1. 외국인 본인인증 지원 여부

가이드 §2에 2차인증 타입으로 `mobile_identification` (휴대전화 실명인증: 이름 + 휴대폰번호)이 있으나, **외국인등록번호** 기반 본인인증 지원 여부가 미명시되어 있습니다. 저희 플랫폼은 발주처/공장 중 일부가 외국인 사업자일 가능성이 있어 중요한 사항입니다.

**답변 요청**:

1. `mobile_identification`이 외국인 휴대폰 (외국 통신사 가입자) + 외국인등록번호 기반 인증을 지원하는지
2. 미지원 시 별도 외국인 본인인증 옵션 (e.g., `foreigner_registration_number`, `passport_verification`) 제공 예정 있는지
3. 외국인 사업자 계약 체결 시 우회 가능한 패턴 (e.g., email + 2차 비밀번호만 사용)

### Q2. Sandbox / Dev 환경 정책

가이드에 dev/staging/prod 환경 분리 정책이 명시되어 있지 않습니다. 신규 가입 시 10건 무료 포인트는 있으나 sandbox 별도 URL/key 정책 미상.

**답변 요청**:

1. 다음 중 어느 패턴인지 확인 부탁드립니다.
   - (a) 단일 base URL + **별도 dev API KEY** (월 사용량 면제 또는 무료 한도 N건)
   - (b) **별도 sandbox URL** (예: `app-sandbox.ucansign.com/openapi`)
   - (c) prod key로 dev/test 진행 (포인트 차감 risk)
2. (a) 또는 (b)일 경우 **dev key 발급 절차** + **무료 한도** 명시 부탁드립니다.
3. dev 환경에서 사용한 계약서가 실제 법적 효력을 가지는지 (저희 측에서 mock template + mock 참여자로 test 시) 확인 부탁드립니다.

### Q3. Webhook signing + payload + retry policy

가이드 §8에 4 events (`sign_creating` / `signing_canceled` / `signing_completed` / `signing_completed_all`) payload는 명시되어 있으나 **signing secret 발급 + 검증 방식 + retry policy가 미명시**입니다.

**답변 요청**:

1. **Signing secret** 발급 + 검증 방식 (HMAC-SHA256? 어느 헤더에 서명 첨부? 예: `X-UCanSign-Signature`)
2. **수신 endpoint whitelist** 등록 절차 (개발자센터에서?)
3. **Retry policy** (실패 시 재시도 횟수 + 간격)
4. **이벤트 순서 보장 여부** (`signing_completed` 후 `signing_completed_all`이 같은 webhook session 내 보장?)

### Q4. audit-trail = 한국 전자문서법 §4의2 감사추적인증서?

가이드의 `GET /openapi/documents/:docId/audit-trail`은 명시되어 있으나 "한국 전자문서법 §4의2 감사추적인증서" 라는 법적 명칭은 미명시입니다. 분쟁 발생 시 법적 증빙으로 활용해야 하므로 중요합니다.

**답변 요청**:

1. audit-trail PDF가 한국 전자문서법 §4의2 요건 (감사추적인증서) **법적 효력** 충족하는지 명시적 확인
2. audit-trail에 포함되는 정보 (서명 시각 + IP + userAgent + 인증 방식 등)
3. PDF의 무결성 (e.g., 디지털 서명, timestamp authority) 검증 가능 여부

## 5. Tier 2 — SHOULD 문의 (Phase 3 design 영향)

다음 4개 항목은 Phase 3 본격 진입 후에도 가능하지만, 사전 확인이 있으면 설계 수정 비용을 줄일 수 있습니다.

### Q5. 가격 / 사용량 정책 상세

가이드에 명시된 사항: 계약 1건당 **100원** + 신규 가입 시 **10건 무료 포인트** + API 이용료 0원.

**답변 요청**:

1. **포인트 충전 단위** (1만원당 100건? 그 이상 단위?) + 최소 충전 금액
2. **월 한도** — 한도 초과 시 정책 (자동 차단 / 추가 과금 / 사전 통보)
3. **결제 주기** — prepaid / 월별 청구 / 분기별 등
4. **계약 기간** — 최소 의무 기간 + 해지 정책
5. **추가 비용** — 알림톡 발송 1건당, audit-trail PDF 발급, 첨부파일 저장, embedding page 호출 등 별도 과금 항목이 있다면 명시
6. **사용량 모니터링 SLA** — `GET /openapi/point/balance` real-time 정확성 보장 여부
7. (참고) 저희는 **2026년 7-12월 베타 운영 기준 월 50-200건 사용량** 예상. 해당 규모에 맞는 요금제 추천 부탁드립니다 (월 5천-2만원 비용 추정).

### Q6. 변경계약서 지원 (FR-5.6 P1)

저희 PRD에 "체결된 계약의 일부 조항 변경" 기능이 P1 우선순위로 정의되어 있습니다. 가이드에 변경계약서 전용 API가 명시되어 있지 않습니다.

**답변 요청**:

1. 다음 중 어느 패턴이 권장되는지 알려주시면 감사하겠습니다.
   - (a) 기존 계약 cancel + 새 계약 생성 (audit-trail 단절 우려)
   - (b) **별도 변경계약서 endpoint** (예: `POST /openapi/documents/:docId/amend`)가 가이드에 누락된 경우 → 별도 문서 공유
   - (c) vendor 측 미지원 — 저희 자체 솔루션 (재계약 + cross-reference) 필요
2. (b)일 경우, audit-trail이 원계약-변경계약을 연결해서 발급되는지

### Q7. integrity_hash (SHA-256) 자체 제공 여부

분쟁 발생 시 PDF 무결성 증빙이 필요합니다. 일부 vendor는 응답 필드로 `integrity_hash`를 제공합니다. UCanSign 가이드에는 명시되어 있지 않습니다.

**답변 요청**:

1. vendor 측에서 계약 완료 시 PDF + audit-trail 결합 SHA-256 해시를 자체 제공하는지 (응답 필드에 있는데 가이드 누락일 수 있음)
2. 미제공 시 저희 측에서 PDF + audit-trail binary concat → SHA-256 자체 계산하는 패턴이 분쟁 시 법적 효력 인정 가능한지 (vendor 의견)

### Q8. Multi-tenant 권한 / 조직 관리 + Locale

가이드상 API KEY는 회원 (조직) 단위로 발급됩니다. 저희 서비스는 발주처/공장 두 그룹이 같은 플랫폼 위에서 계약을 체결하는 구조입니다.

**답변 요청**:

1. **단일 운영 조직 API KEY**로 모든 발주처×공장 계약을 처리하는 것이 정책에 부합하는지 (저희 측에서 계약 row에 발주처/공장 metadata를 customValue로 매핑 가정)
2. 또는 **발주처/공장별 별도 조직 API KEY** 발급이 가능한지 (vendor 측 multi-tenant 지원 여부)
3. **에러 응답 locale** — 항상 한국어인지, 또는 `Accept-Language` 헤더로 제어 가능한지
4. **카카오 알림톡 본문** — KST 시각 표시인지 (vendor 측에서 timezone 변환 처리하는지)
5. (향후) 다국어 계약 처리 시 별도 절차 필요 여부
