## 1. 인사 + 발신처 소개

안녕하세요, 스노우싸인 운영팀 귀중.

저희는 동국대학교 기술창업캡스톤디자인 과목에서 진행 중인 **루트매칭 (Rootmatching)** 팀입니다. 한국 뿌리산업(주조 · 금형 · 소성가공 · 용접 · 표면처리 · 열처리 6대 공정)의 B2B 발주처와 공장 간 매칭 + 전자계약 + 에스크로 결제까지 한 번에 처리하는 SaaS를 개발하고 있습니다.

전자계약 vendor 결정 과정에서 스노우싸인 Public API Guide v1.3을 검토한 결과, PRD 요구사항(전자서명 · 한국 전자문서법 §4의2 감사추적인증서 · integrity_hash · 모바일 알림톡 자체 지원 등) 5/5를 모두 만족하여 **스노우싸인을 정식 채택**하기로 했습니다.

Phase 3(전자계약 통합) 본격 구현 전, 가이드에 명시되지 않은 몇 가지 운영/기술 항목을 확인드리고자 본 문의를 드립니다.

## 2. 통합 컨텍스트 (가이드 v1.3 기준 저희 측 정리)

- **Base URL**: `https://api-snowsign.jtsnowball.com/public`
- **인증**: `X-API-Key` 헤더
- **사용 예정 endpoint**: `/v1/templates/{id}/create-contract` → `/v1/contracts/{id}/send` → status sync → `/v1/contracts/{id}/download` + `/v1/contracts/{id}/audit-certificate`
- **보안 method**: prod default `identity_verification` 양측 + factory 측에 `easy_cert` 옵션 허용
- **통합 형태**: NestJS `contracts/` 모듈 + `SnowsignClient` HTTP wrapper + Prisma `Contract` model + (webhook 또는) `ContractsPoller` Cron 5min

## 3. ⚠️ 보안 사항 — 노출된 API key 폐기 요청

먼저 죄송한 말씀과 함께 보안 관련 처리를 부탁드립니다.

테스트 과정에서 최근 발급받은 API key 일부가 저희 개발 채팅 transcript에 plain text로 노출되었습니다. **저장소(git) · 환경 변수 파일에는 저장되지 않았으나**, transcript 보관 가능성을 고려해 사전 대응을 진행합니다. 노출된 키는 콘솔에서 식별 가능하므로 별도 송부 시 본인 콘솔에서 확인 후 명시 부탁드립니다.

**요청 사항**:

1. 저희 측 콘솔에서 **해당 키 즉시 폐기 (revoke)** + **새 API key 발급** 진행하였습니다 (송부 전 사용자 측 처리 완료 후 본 문구 송부). 추가 조치 필요 시 안내 부탁드립니다.
2. **이전 키 사용 이력 (last accessed timestamp + IP)** 확인 가능 여부 알려주시면 감사하겠습니다. 의심스러운 사용 흔적이 있을 경우 조직 차원의 보안 점검을 진행하겠습니다.
3. 향후 키 노출 방지를 위해 **콘솔에서 별도 dev/prod 키 분리 발급 가능 여부**도 함께 알려주시면 좋겠습니다 (자세한 dev 환경 정책은 §4 Q2에서 별도로 문의드립니다).

## 4. Tier 1 — CRITICAL 문의 (Phase 3 진입 blocker)

다음 3개 항목은 답변 받기 전까지 Phase 3 본격 구현이 보류됩니다. **우선 회신 부탁드립니다.**

### Q1. Webhook 지원 여부

가이드 v1.3 전반에 webhook API가 명시되어 있지 않아 두 가지 통합 패턴을 검토 중입니다.

| 패턴       | 설명                                                        | rootmatching 부담                           |
| ---------- | ----------------------------------------------------------- | ------------------------------------------- |
| (a) 지원   | vendor가 status 변경 시 저희 endpoint로 POST                | webhook controller + signing 검증 구현      |
| (b) 미지원 | 저희 측에서 `/v1/contracts/{id}/status` polling (5min cron) | active contracts 양 증가 시 rate limit 우려 |

**답변 요청**:

1. webhook 기능 **지원 여부** (지원 시 별도 가이드 문서가 있다면 공유 부탁드립니다)
2. 지원 시:
   - signing secret 발급 + 검증 방식 (HMAC-SHA256 + 어느 헤더에 서명 첨부?)
   - payload JSON schema (어떤 event type — `sent` / `signed` / `completed` / `cancelled` / `expired` / `rejected`?)
   - retry policy (실패 시 재시도 횟수 + 간격)
   - 수신 endpoint whitelist 등록 필요 여부

### Q2. Sandbox / Dev 환경 정책

가이드에 dev/staging/prod 환경 분리 정책이 명시되어 있지 않습니다.

**답변 요청**:

1. 다음 중 어느 패턴인지 확인 부탁드립니다.
   - (a) 단일 base URL + **별도 dev API key** (월 사용량 면제 또는 무료 한도 N건)
   - (b) **별도 sandbox URL** (예: `api-snowsign-sandbox.jtsnowball.com`)
   - (c) 모든 호출이 실제 사용량 차감 (이 경우 e2e 테스트 비용 우려)
2. (a) 또는 (b)일 경우 **dev key 발급 절차** + **무료 한도** 명시 부탁드립니다.
3. dev 환경에서 사용한 계약서가 실제 법적 효력을 가지는지 (저희 측에서 mock template + mock 참여자로 test 시) 확인 부탁드립니다.

### Q3. 가격 / 사용량 정책

`POST /v1/contracts/{id}/send` 시점에 "월간 계약 사용량 차감"이 명시되어 있으나 구체 단가가 미공개입니다.

**답변 요청**:

1. **단가** — 계약 1건당 비용 (보안 method별 차등 여부 — `password` vs `identity_verification` 등)
2. **월 한도** — 한도 초과 시 정책 (자동 차단 / 추가 과금 / 사전 통보)
3. **결제 주기** — prepaid / 월별 청구 / 분기별 등
4. **계약 기간** — 최소 의무 기간 + 해지 정책
5. **추가 비용** — 알림톡 1건당, audit certificate 발급, bulk download 등 별도 과금 항목이 있다면 명시
6. (참고) 저희는 **2026년 7-12월 베타 운영 기준 월 50-200건 사용량** 예상. 해당 규모에 맞는 요금제 추천 부탁드립니다.

## 5. Tier 2 — SHOULD 문의 (Phase 3 design 영향)

다음 항목은 Phase 3 본격 진입 후에도 가능하지만, 사전 확인이 있으면 설계 수정 비용을 줄일 수 있습니다.

### Q4. 변경계약서 지원 (FR-5.6 P1)

저희 PRD에 "체결된 계약의 일부 조항 변경" 기능이 P1 우선순위로 정의되어 있습니다. 가이드에 변경계약서 전용 API가 명시되어 있지 않습니다.

**답변 요청**:

1. 다음 중 어느 패턴이 권장되는지 알려주시면 감사하겠습니다.
   - (a) 기존 계약 `cancel` + 새 계약 생성 (audit trail 단절 우려)
   - (b) **별도 변경계약서 endpoint** (예: `POST /v1/contracts/{id}/amend`)가 가이드에 누락된 경우 → 별도 문서 공유
   - (c) vendor 측 미지원 — 저희 자체 솔루션 (재계약 + cross-reference) 필요
2. (b)일 경우, audit certificate가 원계약-변경계약을 연결해서 발급되는지

### Q5. Multi-tenant 권한 / 조직 관리

가이드상 API key는 **조직 단위**로 발급되는 것으로 보입니다. 저희 서비스는 발주처/공장 두 그룹이 같은 플랫폼 위에서 계약을 체결하는 구조입니다.

**답변 요청**:

1. **단일 운영 조직 API key**로 모든 발주처×공장 계약을 처리하는 것이 정책에 부합하는지
2. 또는 **발주처/공장별 별도 조직 키** 발급이 가능한지 (vendor 측 multi-tenant 지원 여부)
3. (1)을 권장하시는 경우, 저희 측에서 계약 row에 발주처/공장 metadata를 함께 저장하는 것 외 추가 책무가 있는지

### Q6. 에러 응답 / 로케일

가이드 예시는 한국어 JSON UTF-8입니다.

**답변 요청**:

1. API 에러 메시지가 **항상 한국어**인지, 또는 `Accept-Language` 헤더로 제어 가능한지
2. 모바일 알림톡 본문이 KST 기준 시각 표시인지 (vendor 측에서 timezone 변환 처리하는지)
3. (i18n 향후 가능성) 영어/일본어 등 다국어 계약 처리 시 별도 절차 필요 여부
