# RootMatching 프론트엔드 기능 명세서

- 작성일: 2026-06-02
- 기준 코드: `frontend/src` Vue 3 + Vite 프론트엔드
- 문서 목적: 현재 프론트엔드에 구현된 화면, 사용자 흐름, 상태 관리, 데이터 구조를 기준으로 서비스 기능을 정리한다.

## 1. 서비스 개요

RootMatching은 발주처와 제조 공장을 연결하는 뿌리산업 B2B 수주 매칭 플랫폼이다. 발주처는 제작 의뢰를 등록하고 AI 추천 공장을 확인한 뒤 계약/에스크로/거래 진행/검수/리뷰까지 수행한다. 공장은 공장 프로필을 등록하고 받은 견적 요청을 검토해 견적을 제출하며, 거래 진행률과 작업 상태를 업데이트한다.

### 1.1 주요 가치

- 발주처: 공정, 품목, 수량, 납기, 예산, 상세 요구사항을 입력해 적합 공장을 추천받는다.
- 공장: 실제 가능한 공정, 설비, 생산 가능량, 포트폴리오를 기반으로 수주 기회를 확보한다.
- 플랫폼: 메시지 협의, 전자 계약/에스크로, 진행률 관리, 분쟁 중재 흐름을 제공한다.

## 5. 핵심 사용자 플로우

### 5.1 발주처 견적 요청 플로우

1. 사용자가 `/client/request`에서 프로젝트명, 공정 유형, 품목, 수량, 납기, 예산, 상세 요구사항, 첨부파일을 입력한다.
2. 제출 시 `workflowStore.submitRequest()`로 요청 초안과 파일 목록을 저장한다.
3. `workflowStore.runAIMatching()`이 호출되어 AI 매칭을 수행한다.
4. 완료 후 `/client/matching`으로 이동한다.
5. 사용자는 추천 공장을 필터링하고 하나를 선택한다.
6. `견적 요청` 버튼 클릭 시 선택 공장을 유지한 채 `/contract`로 이동한다.
7. 계약/에스크로 결제를 완료하면 `/transactions/TXN-2026-018`로 이동한다.
8. 납품 검수 승인 시 `/transaction/review`로 이동한다.
9. 리뷰 제출 후 `/dashboard`로 이동한다.

### 5.2 공장 견적 제출 플로우

1. 공장 사용자가 `/factory/onboarding`에서 공장 프로필을 입력한다.
2. `/factory/requests`에서 받은 견적 요청을 검색/필터링한다.
3. 요청 상세(`/factory/requests/:id`)에서 발주 조건과 첨부파일을 확인한다.
4. 견적 금액, 제작 기간, 가능 납기, 결제 조건, 제안 메시지를 입력한다.
5. 제출 시 메시지 스토어에 `[견적 협의]` 메시지를 생성하고 `/messages?message={id}&context=quote`로 이동한다.
6. 거래가 생성된 이후 공장은 `/transactions/:id?mode=factory` 또는 공장 역할 화면에서 진행률과 작업 업데이트를 입력한다.

### 5.3 기업 디렉토리/문의 플로우

1. 사용자가 헤더 검색 또는 `/companies`에서 키워드, 업종, 지역, 규모 필터를 입력한다.
2. `companyStore.filteredCompanies`가 조건에 맞는 기업 목록을 반환한다.
3. 기업 상세(`/companies/:id`)에서 기업 소개, 사업 정보, 인증, 포트폴리오, 연락처를 확인한다.
4. 즐겨찾기를 토글하거나 문의 모달을 열어 제목/내용을 입력한다.
5. 문의 전송 시 메시지 스토어에 새 메시지를 생성한다.

### 5.4 거래/분쟁 플로우

1. `/transactions`에서 전체/발주처/공장 역할별 거래를 필터링한다.
2. 상세 화면에서 거래 단계, 진행률, 업데이트, 납품 파일, 검사 파일을 확인한다.
3. 공장 역할은 진행률, 업데이트 제목, 상세 내용을 입력할 수 있다.
4. 발주처는 검수 승인 또는 문제 신고를 선택할 수 있다.
5. 문제 신고 시 `/disputes/mediation`으로 이동하여 분쟁 유형과 증빙을 제출한다.
6. `/disputes`와 `/disputes/:id`에서 중재 상태, 단계, 타임라인, 요청 해결안을 조회한다.

## 10. UI/UX 공통 기능

- 앱 레이아웃은 사이드바와 상단 헤더로 구성된다.
- 헤더 검색은 기업 디렉토리로 이동하며 `keyword` 쿼리를 전달한다.
- 헤더는 알림/메시지 미읽음 카운트를 표시한다.

- 라우트 해시가 있으면 부드러운 스크롤을 수행하고, 일반 이동은 최상단으로 스크롤한다.
- 공통 컴포넌트로 `AppButton`, `AppBadge`, `ProcessStepper`, `AuthModal`, `NotificationDropdown`을 사용한다.

## 12. 현재 구현 한계 및 주의사항

- 인증 가드가 없어 URL 직접 접근을 막지 않는다.
- 새로고침 시 Pinia 인메모리 상태는 초기화된다.
- 대부분의 데이터는 목 데이터이며 실제 서버 저장이 없다.
- 파일 업로드는 브라우저 `File` 객체 목록 표시 수준이며 서버 업로드가 없다.
- OpenAI API 키가 프론트엔드 환경변수로 노출될 수 있으므로 운영 환경에는 부적합하다.
- 결제/전자서명/에스크로는 실제 PG 또는 계약 API와 연동되어 있지 않다.
- 메시지 답장은 로컬 상태이며 영구 저장되지 않는다.
- 분쟁 신청은 제출 후 목록으로 이동만 하며 실제 케이스 생성 로직은 없다.

| 공장 선택/계약     | 추천 공장을 선택하고 견적 요청을 누르면 계약 화면에 선택 공장 정보가 반영된다.        |
| 견적 제출          | 공장이 견적을 제출하면 메시지가 생성되고 해당 메시지 상세가 열린다.                   |
| 거래 진행 업데이트 | 공장 역할에서 진행률과 업데이트 내용을 입력하면 최신 업데이트가 목록 상단에 추가된다. |
| 검수 승인          | 발주처가 검수 승인하면 리뷰 작성 화면으로 이동한다.                                   |
| 분쟁 신청          | 문제 신고 또는 분쟁 신청 제출 시 분쟁 현황 화면으로 이동한다.                         |
| 기업 문의          | 기업 상세에서 문의 메시지를 전송하면 메시지 스토어에 새 메시지가 생성된다.            |
| 마이페이지 수정    | 기업 프로필 저장 시 사용자 회사 정보와 기업 목록 데이터가 함께 갱신된다.              |
